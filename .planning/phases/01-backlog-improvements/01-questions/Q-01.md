# Q-01 — Index + JSON — garder les deux ?

**Context:** Le mode --markdown génère deux artefacts : INDEX.md (tableau de bord humain listant les questions actives) et `01-QUESTIONS.json` (état canonique machine-readable). Question : les deux sont-ils nécessaires, ou peut-on simplifier ?

**Dependencies:** none

**Options** (tick exactly one box to finalize):

- [ ] (b) JSON seulement — INDEX.md généré à la demande au moment du refresh — réduit les writes, INDEX.md reste disponible
- [ ] (c) INDEX.md seulement — supprimer le JSON, tout l'état est dans les .md files — plus simple mais perd la structure machine-readable

**Discussion:**

> Claude: La distinction clé : le JSON sert au code (refresh lit le JSON pour calculer le DAG et les dépendances), INDEX.md sert à l'humain pour naviguer. Si tu supprimes le JSON, le refresh doit parser tous les .md files pour reconstruire l'état — plus lent mais faisable. Si tu supprimes INDEX.md, le JSON reste mais il n'y a plus de vue d'ensemble facile à ouvrir dans l'éditeur.

> Claude: Voici un comparatif des approches de persistance d'état disponibles dans le contexte GSD :
>
> | Approche | Lisibilité humaine | Machine-readable | Git diff propre | Coût au refresh | Fichiers |
> |----------|--------------------|-----------------|-----------------|-----------------|---------|
> | **JSON + INDEX.md** (actuel) | INDEX.md ✅ / JSON ❌ | ✅ direct | ❌ JSON verbeux | Lecture JSON = rapide | 2 |

> | **Frontmatter dans chaque .md** | ✅ par question | ✅ (parser frontmatter) | ✅ | ⚠️ scan tous les .md | N questions |
> | **INDEX.md avec YAML frontmatter** | ✅ | ✅ | ✅ | ⚠️ tout dans un fichier | 1 |
> | **JSON seulement** | ❌ | ✅ | ❌ | ✅ | 1 |
>
> Observations clés depuis le code :
> - `discuss-question-file.md` lit le JSON à chaque refresh pour connaître `status`, `conversation`, `dependencies` — c'est son seul accès à l'état machine
> - INDEX.md est entièrement regénéré à chaque refresh — c'est déjà un artefact dérivé, pas une source de vérité
> - Le seul endroit où JSON est irremplaçable : les dépendances DAG entre questions (Q-02 bloquée par Q-03, etc.) — une structure arborescente difficile à encoder proprement en Markdown pur
>
> **Mon avis challengé** : le JSON est légitime pour les dépendances et le statut machine, mais YAML serait plus lisible si quelqu'un ouvre le fichier. Pour les projets simples (sans dépendances), le frontmatter dans les .md files est une alternative viable — mais dès qu'on a un DAG, le JSON (ou YAML) centralisé reste plus pratique.
